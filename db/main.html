Introduction

Database Design

The goal of this project is to simplify the process of storing and querying information about library activity. A great deal of information can be gathered without the overhead of modifying the databases used for core library functions. We'll be looking at three specific use cases to start, and discussing how these use cases can be used as examples for addressing other related projects. The three use cases covered in the next sections involve analyzing historical data, collecting data from library activities, and collecting high-volume data.

Analyzing changes in data over time

The Patron Snapshot use case is an example of analyzing existing data that changes over time, but isn't tracked in a historical context in the core database. Read through this section for more information about how to design tables that allow users to analyze historical data.

Collecting library activity data

Many activities take place in the library that don't directly involve patrons checking out items. To analyze the impact of these activities, we need to track them. However, the volume of data is relatively small, and different institutions need to track different things. For this use case, we look at the impact of library instruction on circulation.

Collecting high-volume data

Many types of activities generate too much data to store effectively. The EZproxy software from OCLC is one example of activity data that can grow very rapidly. A single day's activity logs can generate tens or hundreds of thousands of records, amounting to or hundreds of megabytes or more of data per month. This amount of data is too large to store over any time period greater than a few months. This use case example looks at ways to store smaller subsets of data that still provide useful information.
